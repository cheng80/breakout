# 웹 싱글플레이 벽돌깨기 제품 기획서

- 문서 버전: 1.1
- 작성일: 2026-07-21
- 실행 명세: `seed_9aceaf881e5b` / Seed 1.14.0
- Seed QA: 0.88 기준 미달 사용자 승인
- 1차 출시 범위: 스테이지 5개

## 1. 제품 목표

PC와 모바일 브라우저에서 별도 설치 없이 즐길 수 있는 싱글플레이 벽돌깨기
게임을 만든다. 플레이어는 키보드·마우스·터치 중 원하는 입력으로 패들을
조작하고, 장애물과 갤러그식 적 공격을 피하면서 5개 스테이지의 모든 블록을
격파한다. 빠른 클리어와 적 처치로 점수를 높이고 브라우저에 저장된 최고 점수에
도전하는 반복 플레이를 핵심 경험으로 삼는다.

게임의 우선순위는 다음과 같다.

1. 어떤 입력 방식에서도 패들이 손이나 포인터에 가리지 않을 것
2. 규칙과 상태 전이가 예측 가능하며 피격 후 곧바로 이어서 플레이할 수 있을 것
3. 스테이지가 진행될수록 장애물과 적 행동이 실제 전략 변화를 만들 것
4. 시간·적 처치·최고 점수가 재도전 동기를 제공할 것

## 2. 기술 스택과 구현 원칙

| 구분 | 선택 | 사용 목적 |
|---|---|---|
| 언어 | TypeScript | 게임 상태와 규칙의 타입 안정성 |
| 렌더링 | PixiJS 8 WebGL | 게임 객체·효과·HUD 렌더링과 Pointer Events |
| 스타일 | CSS, Media Queries | 중앙 정렬, 화면 적응, 조작 영역 표현 |
| 개발·빌드 | Vite | 로컬 개발 서버와 정적 웹 빌드 |
| 테스트 | Vitest, `tsc --noEmit` | 규칙·충돌·상태 전이 자동 검증과 타입 검사 |
| 입력 | DOM Keyboard Events, PixiJS Pointer Events | 키보드·마우스·터치 통합 |
| 저장 | `localStorage` | 최고 점수와 튜토리얼 완료 상태 저장 |

PixiJS는 렌더링에 집중한다. 게임 규칙, 충돌 결과, 점수, 아이템 타이머와 상태
전이는 렌더러에서 분리한 순수 TypeScript 모듈로 구현한다. 시뮬레이션은 60Hz
고정 timestep으로 실행해 화면 주사율과 관계없이 같은 입력이 같은 결과를 내도록
한다. React, Phaser, 백엔드, 데이터베이스와 서버 동기화는 사용하지 않는다.

## 3. 화면과 조작

### 키보드

- 좌우 화살표 또는 A/D: 패들 이동
- Space의 새 keydown: 패들에 붙어 있는 공 발사
- 레이저 아이템: 별도 키 없이 정해진 간격으로 자동 발사

### 마우스와 터치

게임판 아래에 패들과 분리된 전용 조작 영역을 항상 표시한다. 이 영역에서 시작한
가로 드래그가 패들을 움직이고, 활성 pointer의 pointerup 순간 공을 발사한다.
포인터 좌표는 조작 영역 경계로 제한한다. 첫 pointer만 사용하고 추가 터치는
무시한다. pointercancel 또는 lostpointercapture에서는 발사하지 않고 입력을
초기화한다. 캡처된 pointer는 영역 밖에서 손을 떼더라도 정상 발사한다.

### 반응형 레이아웃

게임 물리 좌표가 화면에 따라 변하지 않도록 논리 게임판을 720×960(3:4)으로
고정한다. CSS Media Queries로 외부 UI 배치만 바꾼다.

- 세로 화면: HUD, 게임판, 논리 높이 160px의 조작 영역을 위에서 아래로 배치
- 가로 화면: HUD를 게임판 옆으로 이동하되 조작 영역은 계속 패들 아래에 배치
- 전체 컨테이너: `100vw`와 `100dvh` 안에서 비율을 유지해 확대·축소하고 중앙 정렬
- 합격 기준: 게임판 비율 오차 0.5% 이하, 중심 오차 각 1 CSS px 이하, 잘림·문서
  스크롤·뷰포트 초과 0

## 4. 튜토리얼

브라우저 프로필에서 처음 게임을 시작하기 전에 튜토리얼을 자동으로 연다.
완료 상태는 `brickBreaker.tutorialSeen.v1=1`로 저장하며, 사이트 데이터가
삭제되면 다시 표시한다. 저장이 실패하면 현재 탭에서만 완료 상태를 유지한다.

HUD에는 접근 가능한 이름이 `튜토리얼`인 `?` 버튼을 항상 제공한다. 플레이어는
이 버튼으로 언제든 안내를 다시 열 수 있다. 튜토리얼을 연 동안 시뮬레이션과 게임
타이머를 정지하고 입력을 해제하며, 닫은 뒤에는 새 입력만 받는다.

튜토리얼은 다음 이미지와 한국어 설명을 함께 제공한다.

1. 키보드 이동과 Space 발사
2. 하단 드래그 영역과 pointerup 발사
3. 하트, 무적, 공 증가, 자동 레이저 아이템

같은 내용을 게임 안 안내와 `docs/tutorial.md`에 제공한다. 모든 이미지에는 대체
텍스트를 넣는다.

## 5. 핵심 게임 규칙

- 한 게임 런은 스테이지 1부터 5까지 순서대로 진행한다.
- 각 스테이지에는 블록, 장애물, 적과 아이템이 등장한다.
- 모든 블록이 파괴되면 적이 남아 있어도 즉시 클리어한다.
- 적 처치는 클리어 조건이 아니며 점수 보너스만 준다.
- 공은 블록과 적에게 피해를 주고 반사된다.
- 공은 장애물에 피해를 주지 못하며 무해하게 반사된다.
- 레이저는 블록 또는 적에게 첫 피해를 준 뒤 소멸한다. 장애물에 닿아도 피해 없이
  차단되며 관통하지 않는다.
- 같은 60Hz tick에 마지막 블록 파괴와 치명적 피격 또는 마지막 공 유실이 함께
  발생하면 스테이지 클리어를 우선하고 하트 차감·게임 오버를 취소한다.

블록과 적의 체력, 공·레이저 피해량은 밸런스 조정값으로 관리한다.

## 6. 스테이지 구성

| 스테이지 | 블록·장애물 | 적 행동 | 플레이 변화 |
|---|---|---|---|
| 1 | 기본 블록 배치와 고정 벽 | 직선 총알, 기본 하강 | 반사와 조작 학습 |
| 2 | 20% 이상 달라진 블록 배치와 좌우 이동 벽 | 교차 사격, 번갈아 하강 | 이동 벽의 위치를 고려한 발사 |
| 3 | 새 블록 통로와 주기적으로 열리는 개폐 게이트 | 적 종류 추가, zigzag 진입 | 열린 통로와 공격 시점 선택 |
| 4 | 1초 이상 경고 후 하강하는 위험 장애물 | delayedDive 또는 feintDive | 패들 이동 경로까지 회피에 사용 |
| 5 | 이동 벽·개폐 게이트·위험 장애물의 복합 패턴 | curveDive와 복합 사격 | 앞선 규칙을 동시에 판단 |

스테이지 2~5는 직전 스테이지 대비 블록 배치·장애물·적 종류·적 행동 중 최소
2개가 달라야 하며, 블록 외 요소를 반드시 하나 이상 포함한다. 스테이지 4와 5는
적 행동 자체가 달라야 한다. 차별화 여부는 동일 seed의 자동 비교로 판정한다.

- 블록 배치: 동일 격자 점유 셀 20% 이상 차이
- 장애물: type이 다르고 600 tick 위치·개폐 trace의 20% 이상 차이
- 적 종류: 발생시키는 공격 event kind 집합 차이
- 적 이동: 600 tick 위치 trace의 평균 거리가 게임판 너비의 10% 이상

색상이나 스프라이트 이름만 바꾸는 변경은 스테이지 차이로 인정하지 않는다.

## 7. 장애물 규칙

장애물은 공의 경로와 레이저 공격을 방해하는 기본 역할과 패들을 위협하는 후반
역할로 나눈다.

- 고정 벽: 공 반사, 레이저 차단
- 이동 벽: 좌우로 움직이며 공의 예상 진로 변경
- 개폐 게이트: 통과 가능한 시간대를 주기적으로 변경
- 위험 장애물: 최소 1초 경고 후 패들 영역을 횡단하거나 하강

고정·이동·개폐 장애물에 공이 닿을 때는 하트가 줄지 않는다. 위험 장애물과
패들이 접촉할 때만 피해원으로 처리한다.

## 8. 적 규칙

적은 갤러그식 대형을 기반으로 총알 발사와 하강 공격을 수행한다. 후반으로 갈수록
zigzag, 지연 하강, 가짜 하강, 곡선 하강처럼 출발 시점과 궤적이 달라진다.

적 총알, 하강 공격, 적 본체, 위험 장애물과 패들의 접촉을 네 피해원으로 정의한다.
보호되지 않은 한 피해 이벤트는 하트 1개만 차감하며, 차감 직후 시작되는 회복
면제가 같은 시점의 추가 피해를 막는다.

## 9. 하트와 피격 후 재개

- 시작 하트: 3개
- 최대 하트: 5개
- 마지막 활성 공 유실: 하트 1개 차감
- 다중 공: 마지막 공이 유실될 때만 하트 차감
- 네 피해원에 보호 없이 접촉: 하트 1개 차감

마지막 공 유실은 피해원이 아닌 별도의 생명 손실 사건이다. 아이템 무적과 피격 후
2초 면제로 막을 수 없다.

하트 차감 후 1개 이상 남으면 다음 순서로 재개한다.

1. 적과 블록의 현재 상태를 유지한다.
2. 공 증가·레이저의 단계와 타이머를 해제한다.
3. 패들을 중앙으로 옮기고 공 1개를 패들에 붙인다.
4. 2초 동안 네 피해원의 피해를 무시한다.
5. 2초가 끝나면 공을 자동 발사한다.

이 2초는 클리어 시간에 포함된다. 하트가 0이면 위 재개 절차 없이 즉시 게임
오버로 전환한다.

## 10. 아이템

| 아이템 | 효과 | 중복·교체 규칙 |
|---|---|---|
| 하트 | 하트 1개 증가, 최대 5 | 다른 효과·타이머를 변경하지 않음 |
| 일시 무적 | 8초 동안 네 피해원 차단 | 다른 모든 아이템과 동시 적용, 별도 타이머 |
| 공 증가 | 활성 공 수 증가 | 레이저를 즉시 교체, 동종 획득 시 12초 리셋·단계 +1 |
| 자동 레이저 | 패들에서 주기적으로 자동 발사 | 공 증가를 즉시 교체, 동종 획득 시 12초 리셋·단계 +1 |

공 증가와 레이저의 최대 단계는 3을 기본값으로 둔다. 공 증가는 단계별 활성 공
상한을 2·3·4개로, 레이저는 발사 간격을 700·500·350ms로 사용한다. 수치는 한
config에서 조정할 수 있지만 교체·리셋·최대 단계 규칙은 유지한다.

스테이지 전환에서는 모든 아이템을 초기화한다. 하트 아이템과 누적 점수는 다음
스테이지로 이어진다.

## 11. 시간과 점수

각 스테이지 시간은 첫 공 발사 tick부터 마지막 블록 파괴 tick까지 측정한다.
피격 후 2초 대기는 포함하고 튜토리얼과 스테이지 전환 시간은 제외한다. 완료한
스테이지의 tick을 저장하고 표시용 초와 총 플레이 시간은 tick 합계에서 계산한다.

- 블록 파괴: 100점
- 적 처치: 300점
- 목표 시간(초): 90, 110, 130, 150, 180
- 스테이지 배율: 10, 12, 15, 18, 22
- 시간 보너스: `floor(max(0, 목표 초×60-클리어 tick)×배율/60)`
- 목표 시간 초과: 시간 보너스만 0점, 별도 감점 없음

모든 점수는 0 이상 `Number.MAX_SAFE_INTEGER` 이하 정수이며 상한을 넘으면 포화
처리한다. 위 점수·시간 값은 밸런스 조정 대상이며 config, PM 변경 기록, 테스트
기대값을 함께 갱신한다.

## 12. 최고 점수 저장

최고 점수는 `brickBreaker.highScore.v1`에 10진 정수 문자열로 저장한다. 현재
점수가 저장값을 초과할 때 즉시 갱신하며 새 게임과 새로고침 후에도 유지한다.

키가 없거나 값이 빈 문자열·음수·소수·비숫자·안전 정수 초과이면 0으로 복구한다.
다른 버전 키는 읽거나 마이그레이션하지 않는다. 저장 API가 실패하면 현재 탭의
메모리 값으로 계속 플레이하되 새로고침 후 보존은 보장하지 않는다. 계정·서버·
다른 기기와 동기화하지 않는다.

## 13. 스테이지 전환과 종료

스테이지 전환 시 누적 점수와 현재 하트만 유지한다. 아이템 효과·단계·타이머,
잔여 적·탄환·드롭·공·레이저와 입력 상태를 모두 제거한다. 전환 중 시뮬레이션,
점수, 스테이지 시간, 생성과 쿨다운을 정지한다.

이전 입력을 해제한 뒤 새 Space 또는 새 드래그 후 pointerup에서 다음 스테이지를
시작하면서 공을 동시에 발사한다. 자동 시작하지 않는다.

게임 오버 화면에는 현재 점수, 최고 점수와 재시작을 표시한다. 스테이지 5 완료
화면에는 다음 정보를 표시한다.

- 최종 점수와 최고 점수
- 총 플레이 시간과 스테이지별 시간
- 적 처치 보너스 합계
- 시간 보너스 합계
- 재시작

게임 오버와 완료 화면의 재시작은 모두 스테이지 1, 점수 0, 하트 3, 아이템 없음으로
시작한다. 유효한 최고 점수와 튜토리얼 완료 상태만 유지한다.

## 14. 수용 기준

1. 최초 자동 표시와 `?` 재열기를 포함한 이미지·문서형 튜토리얼이 접근성 기준과
   일시정지 규칙을 만족한다.
2. 키보드·마우스·단일 터치에서 이동과 발사가 동작하고 취소·다중 터치·영역 밖
   입력이 정의된 규칙대로 처리된다.
3. 세로·가로 화면에서 미디어 쿼리가 적용되고 게임판 비율, 중앙 정렬, 무잘림,
   무스크롤 기준을 만족한다.
4. 정량적으로 차별화된 5개 스테이지, 적 공격, 장애물, 충돌과 클리어 우선순위가
   정의대로 동작한다.
5. 하트, 마지막 공 유실, 네 피해원, 2초 회복 면제와 네 아이템의 중첩·교체·해제가
   정의대로 동작한다.
6. tick 기반 시간, 블록·적·시간 점수, 결과 화면과 로컬 최고 점수의 정상·오류
   저장 경계값이 모두 검증된다.
7. 스테이지 전환, 게임 오버와 완료 후 재시작이 유지·초기화 규칙을 정확히 따른다.

## 15. 검증과 승인

최종 검증은 승인 대상 commit의 detached 전용 Worktree에서 수행한다. 원본 구현
Worktree와 사용자 변경은 검사하거나 수정하지 않는다. 검증은 다음 단계로 나눈다.

1. `npm run verify:prepare`: config, Vitest, 결정론 반복, 타입 검사, Vite 빌드와
   target manifest 생성을 각각 5분 제한 안에 실행한다.
2. 수동 gate: 생성된 commitSha·Seed 버전·buildId를 기준으로 필수 시나리오를
   실행하고 영상·스크린샷·결과와 기획 책임자 승인을 작성한다.
3. `npm run verify:finalize`: HEAD·Seed·buildId를 다시 대조하고 모든 증빙과 승인을
   검증한 뒤 영구 archive에 보존한다.

동일 seed와 입력으로 1000 tick을 두 번 실행한 최종 상태가 같아야 한다. 생성
파일은 임시 경로에서 완성한 뒤 rename하며 중단 후 같은 commit으로 재개할 수 있다.

수동 검증 환경은 다음 세 가지다.

- 데스크톱 Chrome stable 최신 설치 버전: 키보드·마우스
- iPhone 실기기 Safari stable 최신 설치 버전: 터치
- Android 실기기 Chrome stable 최신 설치 버전: 터치

각 환경의 OS명·OS 버전·브라우저명·브라우저 버전·기기 모델을 기록한다. 튜토리얼, 입력, 반응형 레이아웃,
5스테이지, 피해·무적, 아이템, 점수·저장, 전환·종료를 모두 검증한다. 무적 중 네
피해원에 접촉해도 상태가 초기화되지 않고, 종료 후 한 피해에서 하트가 1 감소하며,
2초 안의 추가 접촉이 무시되는 회귀 절차는 세 환경에서 모두 필수다.

영상은 입력·피해·아이템·스테이지 흐름에, 스크린샷은 점수·초기화·결과 화면에,
자동 테스트 결과는 시간·저장·동시 tick 우선순위에 연결한다. 각 증빙에는 testId,
환경·버전, 실행 절차, 예상·실제 결과, 판정, 검증자·시각, Seed 버전, buildId,
commitSha를 기록한다. 동적 결과는 Git 비추적 `.verification-output/`에, 영구
archive는 저장소 밖 Ouroboros 검증 경로에 둔다. 현재 빌드와 식별자가 다르거나
파일이 없고 열리지 않으면 실패다. 구현자 사전 검증 후 사용자(기획 책임자)가
최종 승인한다.

## 16. 범위 제외와 향후 검토

1차 범위에서 제외한다.

- 게임 오버 후 현재 스테이지 이어하기
- 광고 시청 등 보상형 이어하기
- 스테이지 선택과 뉴 게임 플러스
- 계정, 온라인 랭킹, 클라우드 저장, 기기 간 동기화

보상형 현재 스테이지 이어하기는 향후 광고·보상 정책이 확정될 때 별도 기능으로
검토한다. 현재 구조에 미리 구현하거나 추상화하지 않는다.
